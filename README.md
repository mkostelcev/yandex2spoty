# yandex2spoty

Перенос лайкнутых треков из Яндекс Музыки в Spotify.

Написал для себя, когда решил переезжать. Готовые сервисы вроде TuneMyMusic
и Soundiiz бесплатно переносят сотни треков, а не тысячи, и просят делать
плейлисты публичными, потому что ходят в Яндекс парсером. Здесь всё честно
через API (у Яндекса — неофициальный, но давно живущий), бесплатно и без
ограничений на размер библиотеки.

Что важно знать:

* порядок лайков сохраняется — в Spotify библиотека ляжет так же, как была в Яндексе;
* то, что уже лайкнуто в Spotify, не трогается, дубликатов не будет;
* подкасты и аудиокниги из яндексовых лайков пропускаются;
* прогресс пишется в `state.json`, скрипт можно прервать и запустить снова —
  продолжит с того же места;
* всё, что не нашлось, складывается в `unmatched.csv`.

Матчинг не «в лоб» по строке. Названия и артисты нормализуются (выкидываются
`feat.`, скобки с ремастерами и т.п.), артисты сравниваются ещё и в
транслитерации, чтобы Скриптонит сходился со Skryptonite. Плюс сверяется
длительность: если название совпало и трек той же длины (±3 сек), он считается
найденным, даже когда артист на площадках записан по-разному — классический
случай Элджей vs Allj. Порог похожести — константа `MATCH_THRESHOLD` в начале
скрипта, по умолчанию 0.75.

На моей библиотеке (~4700 лайков) не нашлось около 1% — в основном
нишевые русскоязычные артисты, которых в каталоге Spotify просто нет.

## Токены

Нужны четыре переменные окружения, кладутся в `.env` (образец — `.env.example`).

**Яндекс.** Открываем в браузере под своим аккаунтом:

```
https://oauth.yandex.ru/authorize?response_type=token&client_id=23cabbbdc6cd418abb4b39c32c41195d
```

`client_id` здесь — официального приложения Яндекс Музыки, токен выдаётся вам
самим себе. После редиректа в адресной строке будет
`#access_token=y0_...` — это и есть `YANDEX_TOKEN`. Если страница успевает
загрузиться и затереть адрес, токен целиком остаётся в истории браузера.
Живёт около года.

**Spotify.** На [developer.spotify.com/dashboard](https://developer.spotify.com/dashboard)
создаём приложение: Redirect URI — строго `http://127.0.0.1:8888/callback`
(`localhost` Spotify больше не принимает), из API отмечаем Web API. Client ID
и Client Secret из настроек приложения идут в `SPOTIPY_CLIENT_ID` и
`SPOTIPY_CLIENT_SECRET`, redirect uri — в `SPOTIPY_REDIRECT_URI`. Приложение
может оставаться в development mode, для своего аккаунта этого хватает.

## Запуск

Через докер, чтобы не тащить зависимости на машину:

```sh
cp .env.example .env   # заполнить
docker build -t yandex2spoty .
docker run --rm -it --env-file .env -v "$PWD":/app yandex2spoty \
    python transfer.py --no-browser
```

Можно и в venv, если так привычнее: `pip install -r requirements.txt` и
`python transfer.py`.

При первом запуске spotipy напечатает ссылку на авторизацию. Открываем её в
браузере, соглашаемся, нас редиректит на `127.0.0.1:8888/callback?code=...` —
страница не откроется, это нормально, нужен только адрес. Вставляем его
целиком в консоль. Токен закэшируется в `.spotify_token_cache`, дальше
спрашивать не будет.

Перед полным прогоном имеет смысл посмотреть качество матчинга:

```sh
python transfer.py --dry-run --limit 20
```

`--dry-run` ничего не лайкает, только показывает, что нашлось. Полный перенос
нескольких тысяч треков занимает 30–40 минут: между запросами стоят небольшие
паузы, чтобы не упираться в rate limit, а лайки ставятся по одному — при
добавлении пачкой Spotify даёт всем трекам одинаковый timestamp и
перемешивает их между собой.

## Файлы

| файл | что это |
|---|---|
| `state.json` | прогресс: yandex track id → spotify id / MISS |
| `unmatched.csv` | ненайденные треки с причиной |
| `.spotify_token_cache` | OAuth-токены Spotify |

Ненайденные треки при повторном запуске пробуются ещё раз, найденные не
перепроверяются. Так что если поправить матчинг под свою библиотеку, достаточно
просто перезапустить скрипт.
